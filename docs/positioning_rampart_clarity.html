Rampart & Clarity vs. Whizzard

Microsoft's 2026-05-20 announcement of Rampart (pytest-style agent safety tests on PyRIT) and Clarity (pre-implementation decision-support), framed against what Whizzard is actually building.

Bottom line: Rampart and Clarity are complementary to Whizzard, not competitive. They sit at the design and test phases of the agent lifecycle. Whizzard sits at the runtime phase — the part neither Microsoft tool addresses. The announcement validates the broader thesis (agent safety needs dev-workflow-native tooling) without crowding Whizzard's niche.

Where each tool lives in the lifecycle

What each tool actually is

design-time

Clarity

Structured decision-support. Walks engineers through problem clarification, solution exploration, and multi-perspective failure analysis (security / human factors / adversarial). Outputs markdown committed to the repo.

For: enterprise eng + safety teams
Surface: markdown artifacts
Open source: yes (Microsoft GitHub)

test-time

Rampart

Built on PyRIT. pytest-style integration tests for agent safety: cross-prompt injection coverage, statistical (probabilistic) assertions, composable evaluators that inspect tool invocations & side effects, CI/CD-friendly.

For: engineers shipping agents
Surface: Python tests in CI
Open source: yes (Microsoft GitHub)

runtime

Whizzard

Local capability-governance layer. Wraps an agent harness in a hardened, scoped, time-bounded execution sandbox (Docker today) with explicit, human-readable capability grants. Filesystem mounts, network, credentials, MCP surface all gated.

For: solo devs, local agent operators
Surface: CLI + profiles, harness adapters (Hermes, generic shell)
Status: v0.1.0 in pre-release

Implications for Whizzard

1. Niche is unchanged — and arguably reinforced

Neither tool addresses runtime capability containment. Rampart can tell you the agent shouldn't exfiltrate ~/.ssh/ when prompt-injected; Whizzard ensures the ~/.ssh/ mount was never in the sandbox to begin with. That's a defense-in-depth pairing, not overlap.

2. Audience gap is real

Microsoft explicitly aims at "software engineers, product managers, and safety engineers building agentic AI systems within enterprises." Whizzard's docs/vision_and_strategy.md explicitly carves out the opposite: solo devs, local agent enthusiasts, "not initially targeted at enterprise IAM / SOC2." Different buyer, different shape of tool.

3. Two ideas worth borrowing

  • Rampart's pytest pattern for safety regressions maps cleanly onto Whizzard's own policy: a future tests/policies/ suite asserting "profile quarantine denies network", "credential X never appears in sandbox env unless declared" — same shape, applied to capability policy rather than model behavior.
  • Clarity's "markdown artifact per decision" pattern already exists in this repo as docs/decisions.md (an append-only, numbered decision index). The Microsoft framing — failure analysis from multiple perspectives — could be a useful template addition for higher-risk decisions (e.g. new mount types, new adapter trust assumptions).

4. Positioning watch-out

The announcement uses the phrase "bring safety into agent development workflow." Whizzard's elevator pitch ("local capability governance for AI agents") sits adjacent in language but not in function. When Whizzard hits OSS launch, the README and landing copy should explicitly differentiate: runtime containment ≠ safety testing. A one-line comparison ("Rampart tests behavior; Whizzard bounds capability") would prevent confused conflation.

5. Naming collision risk

"Rampart" is firmly in the same conceptual neighborhood as Whizzard (boundaries, containment, defense). "Whizzard" is itself a working name pending a final rename — and the shortlist for that rename should avoid names that read as Microsoft-Rampart-adjacent. The differentiator is capability scoping, not perimeter.
