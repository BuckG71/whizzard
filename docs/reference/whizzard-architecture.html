How it runs

Four lifecycle flows

Launch whiz run · whiz r <preset>
_perform_launch→ resolve profile + mounts + adapter→ safety validation→ write snapshot→ run_shell→ launch sandbox → enforcement monitor
Adjust / Wake mid-session capability change
whiz adjust / wake→ adjust_session→ stop container→ relaunch with new caps→ harness state persists via bind mount
Agent request cooperation layer
whiz_request_mount / _extend→ /run/whiz/requests/→ operator reviews whiz requests→ approve → adjust_session
Enforce time-bounded sessions
monitor_and_enforce→ poll running sandbox→ duration cap / idle timeout hit→ adapter wrap-up → SIGTERM→ record expiry reason

Safety policy

Every mount target falls into one of three tiers

⛔ Hard block — no override
/, $HOME, ~/.ssh, ~/Library, keychains, browser profiles, the Docker socket, the Whizzard config dir.
⚠ Override-required
Broad folders (~/Documents…), cloud-sync roots, parents of registered mounts. Needs --allow-broad-mount + a permitting profile (two gates). Every override is logged.
✓ Allowed
Registered mounts in mounts.json whose paths fall outside the above. The mount list is the permission model.
Cannot be relaxed by future work

Neutral core
