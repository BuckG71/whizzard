Whizzard — Architecture Overview

Local capability governance for AI agents — a harness-neutral containment and policy layer that decides what a contained agent can see and do.

1Core principle

The foundational trust model. It applies to profiles, mounts, adapters, the cooperation layer, and every future integration.

Whizzard controls capabilities.

Agents request capabilities.

Agents do not grant themselves capabilities.

Capability flow is one-way: host → sandbox, never sandbox → host (D-9). Every part of the system is a refinement of this one rule.

2System shape

Whizzard is one product with three internal layers, split across a hard host/container boundary.

Host vs. container

Host machine — control planeContainer — execution plane ("sandbox")
CLI & policy engine, the config registry (profiles.json, mounts.json, harnesses.json, presets.json), session logs, the Discord control bot (planned). The agent runtime / shell, tool execution, and filesystem access — reachable only through registered mounts. One container per session.
The separation is mandatory. The agent lives entirely in the execution plane; everything that decides its capabilities lives in the control plane, out of its reach.

The three internal layers

LayerOwnsMust not know about
Whizzard Core Profiles, mount registry, policy resolution, container execution, network controls, safety validation, session logging, dry-run preview, duration/idle enforcement. Hermes internals, any harness's internals, Discord, MCP specifics. Harness-neutral — the most important rule (D-10).
Harness Adapter Translating one harness's behavior into Whizzard-compatible execution: launch, wrap-up, env/credential injection, capability surfacing, identity at the harness boundary. — harness-specific identifiers stay confined here (D-153).

3Control layering

Whizzard's controls compose in three concentric layers. Each has a different shape, a different enforcer, and a different owner — they never collapse into each other.

Enforcement layer — Whizzard-owned · kernel & Docker enforce

Mounts, network policy, capability drops, image identity, container hardening, session duration. Set at launch via container flags; they exist before the agent does. The agent can never see or modify them — changing them requires a stop+restart (D-27). This is where the safety policy operates.

Behavioral layer — harness-owned

Dangerous-command interception, in-session approval flows, intent gating, /yolo-style bypass. Intent-time, fine-grained decisions that live inside the harness. Whizzard deliberately does not recreate these — duplicating them would add surface the harness already maintains.

Cooperation layer — Whizzard MCP, in-sandbox

An agent-facing API to the host-side capability brain: introspect own constraints, write structured audit entries, and request structural changes. Requests are mediated host-side and applied via stop+restart. It never replaces the enforcement layer — it is a structured window onto it.

Composition rule: the enforcement layer decides what the agent could do; the behavioral layer decides what it will be allowed to attempt within that envelope; the cooperation layer decides what it can ask Whizzard to change about the envelope. Mixing layers breaks the model.

4Safety policy

Safety validation classifies every mount target into one of three tiers. Block by default; require explicit, logged user action to proceed.

TierTargetsGate
Hard block /, $HOME, ~/.ssh, ~/Library, keychains, browser profiles, the Docker socket, the Whizzard config directory. No override. Ever.
Override-required Broad folders (~/Documents, ~/Desktop, …), cloud-sync roots (iCloud, Dropbox, OneDrive), parents of registered mounts. --allow-broad-mount + a profile that permits it (the two-gate model, D-46). Every override is logged to the session record.
Allowed Registered mounts in mounts.json whose paths fall outside the above. None.
Config write-protection invariant. The Whizzard config directory must never be reachable from any agent-writable mount — regardless of what the policy files say. An agent that can write files Whizzard reads can influence its own policies. Enforced at the safety-validation layer; not overridable by any profile or preset.

Override friction is intentional — warnings get ignored, so the gate is "block, require explicit action, log it."

5Harness adapter contract

An adapter is the only seam between harness-specific behavior and the harness-neutral Core. Whizzard decides what capabilities the sandbox has; the adapter decides what runs inside it.

The contract is the HarnessAdapter Protocol in whizzard/adapters/base.py. Two adapters ship today — GenericShellAdapter (a plain shell) and HermesAdapter (the Hermes agent harness). A NanoClaw adapter is planned for v1.0; the core-maintained slate is capped at three (D-155).

Protocol methodResponsibility
start_command()Argv run inside the sandbox as the entrypoint.
container_env()Env vars to inject — incl. harness-config credentials via OneCLI / env fallback (D-134, D-162).
container_mounts()Harness-required host→sandbox mounts (e.g. Hermes HERMES_HOME for state persistence, D-79).
working_dir()Sandbox working directory, or none.
wrap_up()Harness-native graceful shutdown before SIGTERM, bounded by a grace window. No-op for the generic shell.
preflight()Pre-launch checks (e.g. Hermes refuses to launch over a live gateway, D-87).
active_capabilities()Human-readable capability strings for the pre-launch banner.
mcp_env()Env wiring for the in-sandbox Whiz MCP server (D-156).
health_check_command()Optional readiness probe.

Adapters are built from declarative entries in harnesses.json (a versioned schema). Harness-specific identifiers — paths, filenames, env-var names, CLI flags — never appear outside the adapter module (D-153).

6Architectural constants

Five rules that apply to all phases — MVP, v1, beyond — and cannot be relaxed by future work.

One-wayCapability flow is one-way. Agents request; Whizzard grants; agents never self-grant.
Neutral coreWhizzard Core stays harness-neutral. Harness behavior lives in adapters.
Mounts = policyThe mount list is the permission model — visible, named, scoped capability grants.
Config integrityAgent-reachable paths cannot include the config directory. Non-negotiable.
Time-boundedTime-bounded sessions are enforced, not advisory. Duration is a first-class primitive.

How the architecture lands in code today. The CLI binary is whizzard / whiz; one container per session; Python + Typer + Docker.

Key flows

  • Launch — whiz run / whiz r <preset> → _perform_launch resolves profile + mounts + adapter, runs safety validation, writes the snapshot → run_shell launches the sandbox and hands it to the enforcement monitor.
  • Adjust — whiz adjust <session> → adjust_session stops the sandbox and relaunches with new capabilities; harness state persists across the restart via its bind mount.
  • Agent request — the in-sandbox agent calls whiz_request_mount / whiz_request_extend → a request file lands in the sandbox's /run/whiz/requests/ → the operator reviews it with whiz requests → approval routes through adjust_session.
  • Enforcement — monitor_and_enforce polls the running sandbox; on a duration cap or idle timeout it runs the adapter wrap-up and records the expiry reason.
